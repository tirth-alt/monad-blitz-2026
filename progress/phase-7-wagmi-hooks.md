# Phase 7 — wagmi Hooks (frontend integration layer)

**Status:** ✅ Done (written; type-checks against the frontend once wagmi/viem are installed there)
**Files:** `lib/wagmi.ts`, `lib/hooks/useAgentRegistry.ts`, `lib/hooks/useTaskMarketplace.ts`
**Goal:** Give the frontend a clean, typed React API to read/write the contracts and stream
on-chain events — so Person 2's components never touch raw ABIs or addresses.

---

## 1. Conceptual overview (the "why")

The frontend shouldn't know about ABIs, hex encoding, or `parseEther`. It should call
`useAllAgents()` and get a list, or call `postTask("...", "Research", "5")` and get back tx
status. This phase is that **abstraction boundary**: small, purpose-named React hooks built on
**wagmi v2** (the React layer) + **viem** (the low-level Ethereum library).

Everything imports from `lib/contracts/` (the deploy-generated ABIs + addresses), so when we
redeploy, the hooks automatically point at the new contracts — nothing to hand-edit.

Every **write** hook exposes the same transaction lifecycle flags — `isPending` (waiting for the
user to sign), `isConfirming` (tx in mempool), `isConfirmed` (mined) — which is exactly what the
UI needs for loading spinners and the "payment released" flash (PRD Hour 3: "handle transaction
loading states").

## 2. Technical breakdown (the "how")

### `lib/wagmi.ts` — chain + config
- **`monadTestnet`** — a viem `defineChain` built from the deploy-generated `CHAIN` constant
  (id 10143, RPC, MON currency, explorer). One source of truth.
- **`wagmiConfig`** — a minimal wagmi config (injected/MetaMask connector + http transport).
  Person 2 can instead feed `monadTestnet` to RainbowKit's `getDefaultConfig` — the chain
  definition is shared either way.
- **`txUrl(hash)` / `addressUrl(addr)`** — build Monad explorer links for the live feed
  ("click it — opens Monad explorer").

### `lib/hooks/useAgentRegistry.ts`
| Hook | Type | Purpose |
|------|------|---------|
| `useAllAgents()` | read | `{ agents: Agent[] }` for the agent board (one call). |
| `useAgent(wallet)` | read | One agent's profile (for `/agent/[address]`). |
| `useIsRegistered(wallet)` | read | Gate the "register" CTA. |
| `useRegisterAgent()` | write | `register(name, description, tags)` + lifecycle flags. |

Exposes an `Agent` type mirroring the on-chain struct (viem decodes named structs into objects).

### `lib/hooks/useTaskMarketplace.ts`
**Reads:** `useOpenTasks()`, `useAllTasks()`, `useTask(id)`, `useTasksByAgent(agent)` — each
returns typed `Task[]`/`Task` plus `refetch`/`isLoading`.

**Writes** (each returns its action + the shared lifecycle flags via a `useTxWrite()` helper):
| Hook | Action | Notes |
|------|--------|-------|
| `usePostTask()` | `postTask(desc, category, rewardMon, deadlineSeconds?)` | payable — `parseEther(rewardMon)`; deadline defaults to 1 day. |
| `useAssignTask()` | `assignTask(taskId, agent)` | verifier only. |
| `useCompleteTask()` | `completeTask(taskId, resultHash)` | verifier only; releases pay + reputation. |
| `useCancelTask()` | `cancelTask(taskId)` | poster, open task. |
| `useReclaim()` | `reclaim(taskId)` | poster, after deadline. |
| `usePayAgentInstant()` | `payAgentInstant(toAgent, relatedTaskId, amountMon)` | payable — x402 delegation settlement. |

Exports a `TaskStatus` enum (Open/Assigned/Completed/Cancelled/Reclaimed) matching the Solidity
enum order, and a `Task` type.

**Events:** `useWatchTaskEvents(onLogs)` subscribes to **all** marketplace events; each decoded
log carries `eventName` + `args`, so the live feed can render `TaskPosted`, `TaskAssigned`,
`TaskCompleted`, `PaymentReleased`, `InstantPayment`, etc. and link each to the explorer.

### Design choices
- **Async writes** (`writeContractAsync`) so components can `await` and chain UI reactions.
- **`query: { enabled: !!arg }`** on parameterized reads so they don't fire with undefined args.
- **`"use client"`** on the hook files — they use React state/effects, so they're client-only.

## 3. Integration notes for the frontend team

1. **Install deps in the Next.js app:** `wagmi viem @tanstack/react-query` (+ RainbowKit if used).
2. **Path import:** these live at the repo-root `lib/`. With the standard Next.js `@/*` alias,
   import as `@/lib/hooks/useTaskMarketplace` (or relative). Ensure `lib/contracts/index.ts`
   exists — it's generated by `deploy.js`.
3. **Provider setup** (Person 2): wrap the app in `WagmiProvider` (with `wagmiConfig` or a
   RainbowKit config using `monadTestnet`) + `QueryClientProvider`.
4. **tsconfig:** needs `resolveJsonModule: true` (Next.js default) for the ABI JSON imports.
5. **Verifier-only writes** (`assignTask`/`completeTask`) are normally driven by Person 3's
   backend, not the UI — but the hooks exist for testing/manual flows.

> Note: these `.ts` files can't be type-checked in the contracts project (no React/wagmi here).
> They're written against the wagmi v2 API and will compile inside the Next.js app once its deps
> are installed. Any small type tweaks happen during frontend wire-up (Hour 3 integration).
